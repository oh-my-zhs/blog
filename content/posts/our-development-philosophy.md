"빠르게 만든다"는 말은 종종 "대충 만든다"로 오해됩니다. 저희는 그 오해를 깨고 싶습니다. **속도와 품질은 트레이드오프가 아닙니다.** 올바른 방법론과 도구를 선택하면, 빠른 개발이 곧 좋은 개발입니다.

## 원칙 1: 결정 횟수를 줄여라

프로젝트가 느려지는 가장 큰 이유는 기술적 문제가 아닙니다. **결정 피로**입니다.

- "어떤 상태 관리 라이브러리를 쓸까?"
- "폴더 구조는 어떻게 잡을까?"
- "CSS는 뭘 쓸까?"

저희는 이런 결정들을 미리 내려놓습니다. 웹앱은 Next.js + Tailwind + Zustand, 콘텐츠 사이트는 Astro + Tailwind, 백엔드 API는 Node.js + Fastify + Prisma. 이 스택 안에서 최대한 빠르게 움직입니다.

새로운 기술을 도입할 때는 명확한 기준이 있습니다:

1. 기존 스택으로 해결할 수 없는 문제인가?
2. 팀 전체가 일주일 안에 익힐 수 있는가?
3. 3년 후에도 유지보수 가능한가?

## 원칙 2: 배포를 첫날부터

많은 팀이 개발을 완료한 후 배포를 고민합니다. 저희는 반대입니다. **첫날 빈 프로젝트를 먼저 배포합니다.**

```bash
# Day 1: 프로젝트 시작
npm create next-app client-project
cd client-project
git init && git push origin main
# → 즉시 Vercel에 배포, 클라이언트에게 URL 공유
```

이렇게 하면:

- 클라이언트가 실시간 진행 상황을 볼 수 있습니다
- 배포 관련 문제를 초기에 발견합니다
- "언제 볼 수 있어요?"라는 질문이 사라집니다

## 원칙 3: AI를 도구가 아닌 동료로

저희 개발 환경에서 AI는 단순한 코드 자동완성 도구가 아닙니다. Claude Code, GitHub Copilot은 페어 프로그래밍 파트너입니다.

**실제 워크플로:**

```
오전 9:00 - 요구사항 정리 (AI와 함께 엣지케이스 발굴)
오전 9:30 - 아키텍처 설계 (AI가 초안, 인간이 검토)
오전 10:00 - 구현 시작 (AI가 보일러플레이트, 인간이 비즈니스 로직)
오후 3:00 - 코드 리뷰 (AI가 버그 패턴 탐지)
오후 4:00 - 배포 및 QA (AI가 테스트 시나리오 생성)
```

이 방식으로 개발 속도가 3-5배 빨라졌습니다. 하지만 중요한 것은: **AI가 생성한 코드는 반드시 인간이 이해하고 검토합니다.** 이해하지 못하는 코드는 절대 머지하지 않습니다.

## 원칙 4: 테스트는 행동 기반으로

100% 코드 커버리지를 목표로 하지 않습니다. **사용자 행동을 기반으로 한 통합 테스트**에 집중합니다.

```typescript
// ❌ 저희가 피하는 테스트
test("Button component renders correctly", () => {
  expect(wrapper.find("button").text()).toBe("Submit");
});

// ✅ 저희가 작성하는 테스트
test("사용자가 회원가입 폼을 제출하면 이메일 인증 메일이 발송된다", async () => {
  await fillForm({ email: "user@example.com", password: "secure123" });
  await clickSubmit();
  expect(emailService.sent).toContain("user@example.com");
});
```

## 결과: 첫 MVP 4일 만에 배포

이 철학을 적용해서 최근 클라이언트 프로젝트를 스펙 확정 후 4일 만에 MVP를 배포했습니다. 물론 완벽하지 않았습니다. 하지만 실사용자 피드백을 4일 만에 받을 수 있었고, 2주차에는 그 피드백을 반영한 v1.1을 배포했습니다.

**빠른 개발의 진짜 가치는 속도 자체가 아닙니다. 빠른 피드백 루프입니다.**
